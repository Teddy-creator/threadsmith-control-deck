# v0.3.0 Truth Confidence / Recovery Experience v1

## Goal

Make Threadsmith's recovery state easier to trust: when committed truth, context packets, role packets, latest evidence, phase-run state, or git diff disagree, the deck should show one clear confidence signal and one safe recovery action.

## Scope

In scope:

- Add a compact runtime/view-model signal for truth confidence.
- Explain why a project is fresh, watch, or recover in user-facing language.
- Surface stale `current-packet.json`, stale role packets, failed evidence, running/paused phase runs, and dirty-diff mismatch as distinct reasons.
- Keep existing `sync-context`, `run-hygiene`, `create-handoff`, wait, repair, and resume actions as the recovery vocabulary.
- Add focused runtime and Control Deck tests for confidence labels, reason ordering, and recovery copy.

Out of scope:

- Publishing v0.3.0.
- Multi-provider automation.
- External skill execution.
- Modifying or installing the global `$threadsmith` skill.
- Broad UI redesign.
- Native app, hosted backend, embeddings, or RAG.

## Desired Product Behavior

When the user looks at Threadsmith, they should be able to answer:

- Can I trust the current project truth?
- What exactly is stale or contradictory?
- Should I continue, wait, sync context, run hygiene, repair, resume, or create a handoff?
- Is the issue about committed truth, context packet, role packet, evidence, phase-run state, or git diff?

The deck should not require the user to infer this from scattered labels.

## Implementation Plan

### Task 1: Runtime Truth Confidence Model

- [ ] Add a small runtime model, likely near `contextRecovery.ts`, that summarizes:
  - confidence level: `trusted`, `watch`, `recover`
  - primary reason
  - reason list
  - recommended action
  - stale/mismatch surfaces
- [ ] Reuse existing context recovery and gate signals rather than creating a second recovery engine.
- [ ] Keep accepted committed truth as authoritative over stale running or paused phase-run state.

### Task 2: Control Deck Copy and Placement

- [ ] Add the confidence summary to the existing context/recovery surface.
- [ ] Optionally expose a shorter signal on the homepage decision/context card if it can stay compact.
- [ ] Keep the existing visual style and avoid adding new execution buttons.
- [ ] Make the copy concrete: name the stale object and the next safe action.

### Task 3: Tests

- [ ] Add runtime tests for:
  - fresh truth and packets.
  - stale current packet.
  - stale role packet.
  - latest run failed.
  - running/paused phase run.
  - accepted truth requiring next-slice continuation.
- [ ] Add Control Deck render tests for the user-facing confidence label and recovery action copy.

### Task 4: Truth and Closeout

- [ ] Run targeted runtime/control-deck tests.
- [ ] Run workspace tests and JSON/diff hygiene.
- [ ] Write back `.threadsmith` acceptance, evidence, packet, and closeout.

## Done When

- A user can see one clear truth confidence status in the deck.
- The deck says why confidence is degraded without requiring code-like reason decoding.
- The deck recommends exactly one safest recovery action for the current problem.
- Tests cover fresh, stale, failed, running/paused, and accepted-continuation states.
- No global skill mutation, external skill execution, multi-provider automation, release, or broad redesign lands in this slice.


## Stop Condition

Stop when the phase contract is implemented, reviewed, verified, and written back. If the slice starts requiring broader recovery architecture, pause and record that as a follow-up instead of widening scope.
