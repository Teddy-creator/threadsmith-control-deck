# Threadsmith v0.2.0 Context Operating System Implementation Plan

> **For agentic workers:** implement this plan task-by-task. Keep every slice narrow, verify before claiming completion, and update `.threadsmith` committed truth only when the project state really changes.

**Goal:** Turn Threadsmith from a web control deck into a token-aware context operating layer for AI coding workflows.

**Architecture:** v0.2.0 adds a structured context layer on top of existing `.threadsmith` truth. The runtime will generate compact context packets from Project Brief, Current Phase, Acceptance State, repo signals, git diff, and evidence summaries; later slices will derive role-specific packets and expose context health in the control deck.

**Tech Stack:** TypeScript, Node.js, Vitest, Vite/React, existing `@threadsmith/domain`, `@threadsmith/fs-bridge`, `@threadsmith/runtime`, and `@threadsmith/orchestrator` packages.

---

## Product Positioning

v0.1.1 proves Threadsmith can run as a local-first, Codex-only web control deck.

v0.2.0 should prove a sharper idea:

> Threadsmith saves context by making agents read the right project packet, not by replaying long chats or stuffing the whole repo into a prompt.

The public message should be:

- Token-aware by design.
- Role-specific context instead of one giant prompt.
- Project truth over stale chat memory.
- Recovery from committed `.threadsmith` state.
- Codex Desktop enhancement layer, not a replacement for the main conductor surface.

## Non-goals

v0.2.0 must not expand into:

- multi-provider automatic execution
- native desktop app packaging
- hosted backend or cloud sync
- embedding/RAG infrastructure
- replacing Codex Desktop / Codex CLI as the main conversation surface
- automatic writeback for every ordinary chat message

## Implementation Slices

### Slice 1: Context Packet v1

**Goal:** Produce a durable, compact context packet from current `.threadsmith` truth and local repo signals.

**Files:**

- Create: `packages/domain/src/contextPacket.ts`
- Modify: `packages/domain/src/index.ts`
- Create: `packages/runtime/src/contextPacket.ts`
- Test: `packages/runtime/src/contextPacket.test.ts`
- Modify: `packages/fs-bridge/src/fileStore.ts`
- Test: `packages/fs-bridge/src/fileStore.test.ts`

**Packet fields:**

- `packetId`
- `generatedAt`
- `project`
- `goal`
- `currentPhase`
- `scope`
- `acceptance`
- `nextStep`
- `risks`
- `relevantFiles`
- `recentDiff`
- `evidence`
- `sourceRefs`

**Done when:**

- The packet can be derived from an existing `ProjectState`.
- Missing optional signals degrade to empty arrays or honest unknowns.
- The packet can be persisted under `.threadsmith/context/current-packet.json`.
- Unit tests cover normal, missing evidence, and blocked acceptance scenarios.

**Verification:**

```bash
npm run test --workspace @threadsmith/domain
npm run test --workspace @threadsmith/runtime -- src/contextPacket.test.ts
npm run test --workspace @threadsmith/fs-bridge -- src/fileStore.test.ts
```

### Slice 2: Repo Map v1

**Goal:** Generate a lightweight repository map without embeddings.

**Inputs:**

- `package.json`
- workspace package manifests
- top-level directories
- source/test directories
- current git status and changed files

**Output:**

- `.threadsmith/context/repo-map.json`

**Done when:**

- Repo map identifies likely entry points, test locations, packages, and changed files.
- Context Packet v1 can include a small relevant-files list from repo map.
- No embedding, vector DB, or remote service is introduced.

### Slice 3: Evidence Summary v1

**Goal:** Summarize verification and run artifacts without stuffing raw logs into prompts.

**Output:**

- `.threadsmith/context/evidence-summary.json`

**Done when:**

- Evidence summary records command, exit code, status, short conclusion, failure focus, and artifact path.
- Context Packet v1 uses evidence summaries instead of raw stdout.
- Existing smoke and test artifacts remain runtime artifacts, not committed truth by default.

### Slice 4: Role-specific Packets v1

**Goal:** Derive smaller role packets from the main context packet.

**Role packet outputs:**

- planner packet
- executor packet
- reviewer packet
- verifier packet
- closeout packet
- hygiene packet

**Done when:**

- Executor packet focuses on scope, files, and implementation constraints.
- Reviewer packet focuses on diff, requirements, and review blockers.
- Verifier packet focuses on acceptance, commands, and evidence.
- Closeout packet focuses on cleanup, residual risks, docs, and repo state.
- Hygiene packet focuses on freshness, contradictions, stale truth, and handoff needs.

### Slice 5: Threadsmith.skill v2

**Goal:** Make `$threadsmith` a stronger execution protocol backed by context packets.

**Done when:**

- Each role declares required inputs.
- Each role declares allowed `.threadsmith` writes.
- Each role declares forbidden writes.
- Each role has a completion artifact.
- Skill includes explicit truth sync mode.
- Skill includes writeback failure handling.

### Slice 6: Truth Sync / Recovery / Hygiene Triggers

**Goal:** Make stale or interrupted state visible and recoverable.

**Triggers:**

- phase start
- phase end
- before verification
- before PR/merge
- interrupted thread continuation
- `git diff` exists but acceptance is stale
- run is stuck in `running`
- evidence is older than the latest implementation change

**Done when:**

- Threadsmith can recommend sync, continue, verify, repair, closeout, or handoff.
- It does not silently proceed when committed truth contradicts repo state.

### Slice 7: Context Budget Ledger v1

**Goal:** Estimate context size and show whether packets are becoming too heavy.

**Output:**

- `.threadsmith/context/context-budget.json`

**Done when:**

- Budget ledger estimates size by section.
- It flags oversized diff/evidence/history sections.
- It is approximate and transparent, not pretending to be exact tokenizer accounting.

### Slice 8: Control Deck Context Surface v1

**Goal:** Make context health visible without turning the UI into a log viewer.

**UI surface should show:**

- packet freshness
- current packet status
- top relevant files
- evidence summary
- context budget warning
- which role packet is recommended next

**Done when:**

- Existing visual style remains intact.
- Context state is readable at a glance.
- Details live in the appropriate drawer/workbench, not cluttering the homepage.

## Cross-cutting Acceptance Rules

- v0.2.0 remains Codex-only.
- Context Packet generation must be deterministic from committed truth and repo signals.
- Raw logs should stay as artifacts; packets should contain summaries and paths.
- UI should explain context health without requiring users to understand internal schema names.
- Any accepted state must still pass review, verification, and closeout gates.
