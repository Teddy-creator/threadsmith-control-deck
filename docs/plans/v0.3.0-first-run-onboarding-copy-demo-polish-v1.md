# v0.3.0 First-run Onboarding / Copy Slimming / Demo Polish v1

> **For agentic workers:** This slice is about making Threadsmith easier to understand on first contact. Keep the product surface practical and compact. Do not widen into release publishing, multi-provider execution, external skill invocation, native app work, or broad visual redesign.

## Goal

Make Threadsmith's first-run and demo path feel understandable to a new developer: the deck, README, usage guide, and demo truth should explain what Threadsmith does, what it does not do, how local skills are discovered, and what the next safe action is without forcing the user to read internal planning history.

## Scope

In scope:

- Review the first-run path from a fresh user perspective: README top section, usage/configuration guide, launch messages, demo project truth, and Control Deck copy that appears before real workflow execution.
- Slim repeated or internal-facing copy that makes Threadsmith feel heavier than it is.
- Clarify the boundary between Threadsmith as workflow kernel / control deck and the user's conductor chat surface.
- Clarify that External Skill Adapter v1 discovers and routes local Codex skills, but does not autonomously execute arbitrary external skills.
- Make demo/example `.threadsmith` truth align with current v0.3.0 capabilities and wording.
- Keep existing Control Deck visual style; only adjust copy or small information placement if necessary.
- Add targeted tests only where copy/model changes affect runtime or deck rendering.

Out of scope:

- Publishing or tagging v0.3.0.
- Multi-provider automation.
- Real external skill execution.
- Modifying or installing `/Users/cloud/.codex/skills/threadsmith/SKILL.md`.
- Native desktop app, hosted backend, embeddings, or RAG.
- Turning the Threadsmith web UI into the primary chat or command surface.
- Broad visual redesign.

## Desired Product Behavior

A new developer should be able to answer these questions in a few minutes:

- What is Threadsmith?
- Do I still talk to Codex / Claude / my conductor chat directly?
- What does the deck monitor?
- What does `$threadsmith` write back?
- What is automatic today, and what is only routed / recommended?
- What should I do after launching the project for the first time?
- How do I tell whether the current truth is stale, trusted, or needs recovery?

The answer should be discoverable from product-facing surfaces, not from old implementation plans.

## Done When

- README and first-run docs explain Threadsmith in product-facing language.
- First-run/deck copy clearly separates the control deck, conductor chat, `$threadsmith` writeback, skill discovery, and fallback behavior.
- Demo/example truth reflects current v0.3.0 capabilities without overstating automation.
- The Control Deck retains its existing visual style and does not become more cluttered.
- Verification evidence is recorded for every changed surface.
