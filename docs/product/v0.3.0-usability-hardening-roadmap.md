# v0.3.0 Usability Hardening Roadmap

## Decision

Do not publish `v0.3.0` yet. Treat the current branch as an internal release-candidate baseline and continue hardening the product until a new user can understand, configure, and operate Threadsmith without reading internal planning history.

## Product Goal

Threadsmith should feel like a practical workflow kernel for AI coding projects, not just an impressive state schema. The next hardening pass should close the gap between "Threadsmith can describe the workflow" and "a developer can actually rely on it every day".

## What Still Needs Work

### External Skill Adapter v1

Connect local Codex skills to Threadsmith's routing system. Threadsmith should discover skills, infer capabilities, explain availability, and route roles to adapters when safe. Built-in mini protocols remain the fallback.

### Skill Discovery / Health Check

Scan local and project skill directories, parse `SKILL.md` metadata, classify likely capabilities, and report missing / stale / unsafe / fallback-only states.

### Project-level Skill Configuration

Allow each project to pin preferred skills per capability or role. Discovery can recommend mappings, but committed project defaults should make routing stable and reviewable.

Current implementation status:

- `.threadsmith/skill-routing.json` stores project-level route preferences, disabled adapters, discovery provenance, and notes.
- Runtime route resolution can honor capability-wide preferences such as `debug -> systematic-debugging`.
- Runtime route resolution can honor role-specific preferences such as `verifier + verify -> independent-verification`.
- Disabled adapters remain visible but route through the built-in fallback protocol.
- This is still routing metadata only; Threadsmith does not execute external skills automatically in this slice.

### Route Explanation

Every route should be explainable: why this skill, why fallback, what safety warning exists, and what the selected built-in protocol will still enforce.

### All-Codex Autopilot Usability

Reduce repeated user "continue" prompts for ordinary phases while preserving hard stops for risk, scope expansion, missing evidence, writeback failure, and user decisions.

### Truth Confidence Surface

Make the deck and packets answer whether `.threadsmith` truth is fresh, whether git diff and evidence match the acceptance claim, and whether role packets are stale.

### First-run Onboarding

For new projects, guide the first `$threadsmith` invocation through project brief, current phase, acceptance state, and next best step generation instead of leaving the user to infer missing truth.

### Command Bridge Stability

Keep deterministic smoke, real diagnostic, and production run as separate modes. Real Codex CLI diagnostics must be opt-in because they can depend on local auth, state, network, and selected project mutability.

### Copy / Documentation Slimming

Make README, deck copy, and release notes product-facing. Keep internal engineering history in plans and closeouts, not in the first-run user path.

### Configuration Visibility

Expose provider, skill adapter, fallback, auto-execution support, and safety status in a compact surface so users can trust what Threadsmith is about to do.

### Recovery Experience

When interrupted, failed, stale, or contradictory, Threadsmith should propose one safe recovery action and record why it chose that path.

### Windows Experience

Keep launchers, paths, PowerShell notes, and troubleshooting aligned with macOS behavior before public release messaging implies cross-platform smoothness.

### Demo Project

Provide a small demo repo or scripted project that shows Threadsmith driving a real task in roughly five minutes.

## Release Readiness Bar

`v0.3.0` should not be public until these are true:

- External Skill Adapter v1 can discover and route local Codex skills safely, with built-in fallback.
- A fresh project can bootstrap into usable `.threadsmith` truth without manual JSON editing.
- The deck can explain current route / skill / fallback / truth freshness in user-facing language.
- Deterministic verification remains green.
- README onboarding matches actual product behavior.

## Non-goals For This Hardening Track

- Fully automated multi-provider execution.
- Real external skill invocation that bypasses Codex's skill mechanism.
- Native desktop app packaging.
- Hosted backend, embeddings, or RAG.
- Replacing the user's conductor chat surface.
