# Threadsmith v0.3.0 Research Plan: Skill Orchestrator and Mini Protocols

## Threadsmith Decision

Threadsmith v0.3.0 should not become a giant skill that swallows every other vibe-coding skill. It should become a workflow kernel:

- Committed `.threadsmith` truth remains the durable project state.
- Built-in mini protocols provide reliable default behavior when no external skills are installed.
- External skills become optional capability adapters with explicit inputs, outputs, freshness rules, and fallback behavior.
- Context packets stay small and role-specific instead of replaying long chat history.
- The stable installed `$threadsmith` skill controls Threadsmith's own development; repository skill/protocol changes are source changes until reviewed, tested, and intentionally installed.

## Goal

Define the v0.3.0 architecture for:

- Skill Orchestrator v1
- Built-in Mini Protocols v1
- External Skill Adapter Contract v1
- Research-backed Harness Engineering foundations
- Self-hosting safety for developing Threadsmith with Threadsmith

This document is a planning artifact, not an implementation patch.

## Current Baseline

v0.2.0 shipped the Context OS baseline:

- committed `.threadsmith` truth
- Context Packet v1
- Repo Map v1
- Evidence Summary v1
- Context Budget Ledger v1
- Role-specific Packets v1
- Threadsmith skill v2
- Truth sync and recovery/hygiene triggers
- Control Deck context surface
- Context packet regeneration

v0.3.0 should build on that baseline by deciding which workflow protocol to run next, not by adding more undifferentiated prompt text.

## Scope

In scope:

- Capability registry for internal mini protocols and optional external skills
- Routing rules that choose built-in mini protocol vs external skill adapter
- Input/output contracts for brief, plan, implement, review, verify, closeout, recover, and handoff work
- Context budget rules that keep packets compact and role-specific
- Tests that prove routing, fallback, and freshness behavior
- Self-hosting safety model for developing Threadsmith without corrupting the active stable controller

Out of scope for this planning slice:

- Product code implementation
- Installing or overwriting the global `$threadsmith` skill
- Multi-provider automatic execution
- Native desktop shell
- Hosted backend, embeddings, or RAG
- Turning the Control Deck into the primary chat surface

## Mapping to Threadsmith Objects

### Schema

Add or extend committed/runtime objects only when they answer a durable question:

- `project-brief.json`: what the project is trying to achieve and which constraints matter.
- `current-phase.json`: which protocol is allowed right now and what must stop it.
- `acceptance-state.json`: which gates are pending, blocked, or passed.
- `project-supervision.json`: which logical roles are active and what each owns.
- New candidate: `skill-orchestrator.json`, committed only if it describes stable project defaults such as enabled adapters and fallback policy.
- New candidate runtime artifact: `.threadsmith/protocol-runs/<run-id>.json`, for individual mini protocol or adapter runs.

### Protocol

Each protocol route should answer:

- Which role owns the next move?
- Which mini protocol or adapter is selected?
- What packet does it receive?
- What evidence or artifact must it produce?
- What state transition is allowed after it finishes?
- What fallback runs if the external skill is missing, stale, or incompatible?

### Tests

v0.3.0 should add tests before broad implementation:

- Routing selects a built-in mini protocol when no external skill exists.
- Routing selects an external skill adapter only when the adapter contract is satisfied.
- Missing external skills do not block common workflows.
- External adapter output cannot mark verification passed without evidence.
- Role packets stay smaller than the main context packet.
- Stale phase or stale acceptance forces sync/recover before execution.
- Self-hosting mode never writes repository development skill files into the global installed skill path.

## Built-in Mini Protocol Inventory

Built-in mini protocols should be intentionally small. They are not full replacement skills; they are the reliable fallback kernel.

| Mini protocol | Purpose | Minimal input | Required output |
| --- | --- | --- | --- |
| `brief` | Turn fuzzy request into a task contract. | user goal, project brief, constraints | goal, scope, non-goals, done when, verification |
| `plan` | Turn accepted brief into steps. | current phase, acceptance state, repo map | implementation plan, stop condition, verification commands |
| `debug` | Investigate failure without guessing. | failure symptom, evidence summary, changed files | root-cause hypothesis, reproduction path, repair plan |
| `review` | Separate implementation from critique. | diff summary, acceptance claim, risks | findings, residual risks, ready/blocked decision |
| `verify` | Require evidence before acceptance. | done when, commands, evidence refs | pass/fail result, exact evidence, missing checks |
| `closeout` | Clean up and record durable next step. | accepted claim, diff, docs impact | cleanup result, residual risks, next slice |
| `handoff` | Compact continuation state. | current truth, latest evidence, open risks | text-only handoff or context packet refresh |
| `recover` | Re-anchor stale or interrupted work. | committed truth, context packet, git state | stale facts, repaired truth or next safe action |
| `research` | Ground design in external sources. | research question, project constraints | source list, takeaways, Threadsmith mapping |

## External Skill Adapter Contract

External skills are optional enhancements. Threadsmith should not require a user's local skill library to match the maintainer's machine.

An adapter should declare:

- `id`: stable skill or adapter name.
- `capabilities`: brief, plan, debug, review, verify, closeout, handoff, recover, research, frontend, docs, etc.
- `entry`: skill name, file path, CLI command, or human instruction.
- `inputContract`: required fields from Threadsmith truth/context.
- `outputContract`: required artifact shape and allowed state transitions.
- `freshness`: when the adapter is compatible with current phase/acceptance.
- `fallback`: built-in mini protocol to run if unavailable.
- `safety`: files or state it is forbidden to mutate.

Adapter output should never directly override committed truth. It should produce a proposal or evidence artifact that the owning Threadsmith role can accept or reject.

## Self-Hosting Safety Model

Developing Threadsmith with Threadsmith creates a bootstrapping hazard: the tool being changed is also managing the change.

Safety rules:

- The installed global `$threadsmith` skill is the active controller.
- Files under `codex/skills/threadsmith/` are source code, not the active controller.
- Repository skill changes must be reviewed and tested before manual installation.
- Planning, implementation, and verification should record whether they used the installed skill or repository skill source.
- No planning slice may write to `/home/user/.codex/skills/threadsmith` unless the explicit task is "install/upgrade skill" with backup and verification.
- Control Deck truth should describe the current source phase, not pretend the active installed controller already has unreleased behavior.

## Harness Engineering Principles for v0.3.0

Threadsmith should treat the harness as product code:

- State is explicit and versionable.
- Context is budgeted and role-specific.
- Role transitions are constrained.
- Verification requires evidence, not confidence.
- Recovery starts from committed truth, not chat memory.
- External skills are routed by capability and contract, not copied into a mega prompt.
- The operator can inspect what happened without reading a giant transcript.

## Done When

v0.3.0 planning is accepted when:

- `.threadsmith` committed truth points to v0.3.0 planning, not v0.2.0 release hardening.

## Suggested First Implementation Slices

1. `skill-orchestrator-schema-v1`
   Define adapter registry types, mini protocol IDs, routing decisions, and fallback policy in runtime/domain tests.

2. `mini-protocols-v1`
   Implement small built-in protocols for brief, plan, verify, closeout, recover, and research as deterministic templates over committed truth.

3. `adapter-contract-v1`
   Add optional external skill adapter declarations without requiring the external skill to exist locally.

4. `context-routing-budget-v1`
   Ensure route packets are small, role-specific, and fail freshness checks when phase/acceptance drift.

5. `self-hosting-safety-v1`
   Add tests and docs proving repository skill source changes cannot silently mutate the installed global controller.

6. `deck-orchestrator-surface-v1`
   Show selected route, fallback status, adapter availability, and next protocol evidence without turning the deck into a chat surface.

## Open Questions

- Should `skill-orchestrator.json` be committed truth, project preference, or derived runtime state?
- Should adapters be project-local, global, or both?
- How strict should context budget limits be for mini protocol inputs?
- Should research mapping be a release gate for all future major protocol changes?
- Should protocol runs produce markdown artifacts, JSON artifacts, or both?
